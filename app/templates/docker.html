Containerization vs Virtualization

IBM Blog: Cloud Computing

We've all seen the benefits that come with cloud computing; even my University has switched to it. In this space, there are two major contenders whe it comes to cloud native development and DevOps: Virtualization and Containerization. So, what are they?

Virtual Machines are probably the most known out of the two; it's essentially running an OS (or multiple) within another one. Virtual machines are great and are comparable to emulating physical computers. This however, comes with the drawback of having to start the OS every time as if it were an actual computer, causing major delays when dealing with big centers. On the other hand, Containers don't have said limitation. This is because they virtualize operating systems instead of systems. It shares the kernel, CPU, Memory, & space of the main system. Efficient containers also contain only what's required to run the desired application, so they tend to be on the safer side as well. The Cloud Computing world is leading towards container, with eyes mainly on Docker and Kubernetes.

Images

Docker Docs: Images

Images are read-only instructions for building containers. For example, Docker brings up the possibility of having an ubuntu-based image that installs the Apache web server, your app, as well as all the dependencies for running said app.

As seen above, one can create their own images. On top of these, there are plenty of official images out there as well as those created by other users. Using images, containers are created (which are runnable). Containers tend to be independent of your host machine and other containers unless explicitly stated.

Docker Repositories

Docker Docs: Repositories

In the same way that you'd want a repository for your app code, you also want one for your containerization projects.

To create a Docker repository, just head over to the Docker website (known as DockerHub), click on "Repositories" on the navbar, and name it appropriately. To push to a repository, one types the following into the terminal: "docker push DOCKER_USERNAME/REPO_NAME:TAG". Make sure the repository name matches the one you gave it earlier, and give it any tag you want (latest is preferred). You can then pull from the repository using docker pull {repoName}.

Project Creation via the Terminal

Docker Docs: Tutorial

The preferred method by many is to use Docker/DockerCompose Files within IDEs and start the project there, but you'll learn the alpha method here. In this scenario, I'll basically be giving a short version of Docker's second step out of the 10 they have for getting started.

  1. Download the app from GitHub. You'll find it in the link above.
  2. Within the same directory as the app, create a file with no extension named Dockerfile.
  3. Be sure to navigate to the project directory with using your terminal.
  4. Run the following within your terminal to build a container image: docker build -t getting-started .
  5. Start an app container and route port 3000 of your local system to that of the container by: docker run -dp 3000:3000 getting-started
  6. In your preferred browser, head over to http://localhost:3000 (or type localhost into the browser).
  7. You should see the web app on your browser!

My Terminal Commands

Docker Docs: Commands

I've got my list of useful methods I use here and there. Here are some of them:

General

  1. docker version: Displays docker version information
  2. docker build: Build an image from a Dockerfile
  3. docker commit: Create image from change in container.
  4. docker history: Show image history
  5. docker image{s}: List/Manage Images
  6. docker info: Display system-level information
  7. docker {push/pull}: Repo commands; similar to Git
  8. docker rm{i}: Remove containers or images
  9. docker stop: Stop running containers
  10. docker volume: Manage docker volumes

M1 Specific : Multi-Platform Setup

  1. docker buildx ls: List current builder instances
  2. docker buildx create --name BUILDER_NAME --use: Create a builder and use it
  3. docker buildx inspect --bootstrap: Make sure it works
  4. docker buildx build --platform linux/amd64,linux/arm64,linux/arm/v7 -t USER_NAME/CONTAINER_NAME:latest --push . : Building & Pushing
