SOLID Principles: The Calculator Program

Single Responsibility

Digital Ocean: Definition

Just like Inheritance, the Single-Responsibility Principle is one that is fairly self-explanatory. To sum it up nicely, the principle states that each class should do one thing; and do it well. This way, changes in one place can impact everything in a good way. Troubleshooting will also become a breeze as there's only one place where a specific thing could go wrong. Just taking a look at how the Calculator project's file system is set up shows how it's following this principle. Every major thing is its own class and it does its function well.

Open-Closed

Digital Ocean: Definition

The Open-Closed Principle is also fairly simple; though it is not as transparent as the last. This principle states that completed classes should be open for extension but not for modification, as you don't want to drastically change how it behaves. This means that adding some sort of new feature is allowed as long as it follows the principles and isn't changing any functional behaviors. The code is open to additional functionality but closed to major shifts. With that being said, let's take the example of the code being shown below. If I wanted to implement some sort of new feature within that specific method, I have to do so in a way that doesn't touch or logically change that code, as that method is essential in creating objects for the factory mentioned earlier. A new feature that easily comes to mind is adding some sort of check to ensure that the values being passed as the tuple_list argument are actually numbers, as this adds functionality without taking any away.

Liskov Substitution

Digital Ocean: Definition

The Liskov Substitution principle extends on the Open-Closed Principle and takes aspects of Inheritance into account. This principle focuses more on the behavior of a superclass (parent) and its children, stating that parent objects should be replaceable with their children without breaking the application. As stated in the Inheritance section, child classes should depend on their parent; not the other way around. To get into specifics, this means that overridden methods within the child class should have the same method signature and behave on a strictness-level comparable to that of the parent or the program may throw an exception when dealing with the parent and its children. As seen below, the Calculator parent class has a function that returns the argument converted to a tuple of floats. If the Addition, Subtraction, or other subclasses wanted to modify it, they should do so in a way that doesn't compromise its original usage as that is how you'd follow th principle of Liskov Substitution.

Interface Segregation

Digital Ocean: Definition

The Interface Segregation Principle is similar to the principle of Single Responsibility in the sense that it is meant to reduce widespread issues and enforce organization by separating code by function and features. However, this principle is specific to Interfaces and states that overpopulating or adding more and more to one interface whenever a new feature is requested is bad practice as it leads to a mess and polluting the original interface; it's better to create a new one bearing the new functionality. Although the image below more accurately describes the Single-Responsibility Principle as the add method in the Calculator class makes use of methods from other classes with very specific purposes, you can imagine how it would be for this principle based on that.

The image above shows two separate classes, Multiplication and Subtraction, that have a method named getResult() with the exact same header. The method for Multiplication multiplies before returning and the behavior for Subtraction's getResult() method should be obvious. That being said, the compiler would be able to discern which getResult() is being called depending on which class the object being used to access the method is an instance of.

Dependency Inversion

Digital Ocean: Definition

The idea of the Dependency Inversion principle is not a hard one to grasp as any programmer who has dipped theirs toes in Object-Oriented Programming has done this; at least unintentionally. This principle states that the High-Level modules in the code that provide abstract logic should not be impacted by later created low-level modules that include new features. In order to do this, you (and the low/high-level modules) depend on abstractions in order to split their dependence; hence the name of the principle.
