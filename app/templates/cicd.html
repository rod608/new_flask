Continuous Integration & Deployment

GitHub Blog: CI/CD

CI/CD Is a DevOps tactic that makes use of automating. In this case, these techniques are invoked via the usage of automated testing with GitHub Actions and eventual deployment to Heroku.

Continuous Integration involves the practice of merging changes to the main branch as often as possible (within reason) by running automated tests onto the branch upon certain conditions. Further down the line comes Continuous Deployment, where code that happens to pass all the steps/checks successfully is released to the public automatically (in this case, your Heroku app is updated).

GitHub Actions & Docker Images

GitHub Actions: Features

So, I've got GitHub Actions configured in a way that allows for continuous integration and deployment. GitHub Actions looks for the .yml file within the workflows directory and follows that, similarly to the way that docker looks for Dockerfiles. The first thing one does in the .yml file is give it a name at the top. Afterwards, you specify the conditions for jobs to active (on push or pull requests of the master, in this case). Then, you assign some jobs. In this case, I've assigned jobs for setting up Ubuntu and running tests, building the Docker containers/updating the repository, and deploying to Heroku. The passwords to my Docker account and the access token for my Heroku app have been passed in as GitHub secrets. My development workflow is only triggered upon a pull request, and it serves the purpose of checking if the code should be merged. It won't update a Docker repository but will deploy to the development Heroku app, so one is given the chance to review the site there if needed. The production workflow is triggered upon a git push, and it both updates the associated Docker repository and deploys to the production Heroku app, which is the one that "customers" would see.
