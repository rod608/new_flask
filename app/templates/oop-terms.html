Object-Oriented Programming: Glossary

Class-Related Definitions

  • Class: The blueprints for objects. Properties, methods, and constructors bundled into one unit.

  • Instance Data: The properties/characteristics of said classes.

  • Constructors: Defines how a class is instantiated. Typically initializes or sets the properties to default values.

  • Methods: Functions within the class that manipulate the properties or perform actions related to the class. They define the behavior of the class.

    • Static Methods: Tied directly to the class; not objects. Static properties/methods can only interact with other things declared as static, and can be called just by using the class's name.

Object-Related Definitions

  • Object: An instance of the class. It has all the properties and methods defined by the class and is its own entity.

  • Instantiation: Creating a new instance of the class, or an object. An object of a certain class is an instance of said class.

  • Constructors: Defines how a class is instantiated. Typically initializes or sets the properties to default values.

  • Methods: Functions within the class that manipulate the properties or perform actions related to the class. They define the behavior of the class.

The Four OOP Principles

  • Encapsulation: The fact that we can bundle properties and methods into units called classes. Each instance of the class is also its own unit.

  • Abstraction: Purposely not showing certain properties or methods in classes in order to improve organization and ease of use. The access modifier keywords come into play here a lot.

  • Inheritance: The process in which a subclass extends from a superclass. The child class inherits all properties and methods from the parent while having the potential to add its own. One parent can have multiple children, but children can only have one direct parent.

  • Polymorphism: Taking many forms. Refers to the fact that different classes/objects may have methods with the same signature and yet the compiler knows which to invoke depending on the instance of said object.

The Four SOLID Principles

  • Single-Responsibility: Each class should do one thing and do it well.

  • Open-Closed: Code should be open for extensions, but closed for impactful modifications.

  • Liskov Substitution: In reference to Inheritance, the parent clas should be able to be substituted for the child without breaking the code. Children depend on parents.

  • Interface Segregation: Similar to the Single-Responsibility Principle, each interface should do their own thing well. New interfaces should be created for new features.

  • Dependency Inversion: Majorly impactful code should not be impacted by small, feature-like code. Dependency should be split by having both be derived from one abstraction.
