Object-Oriented Programming: Glossary

GeeksForGeeks: Procedural vs Object-Oriented Programming

Class

Brilliant.org Article

When you hear the word "class" in reference to object-oriented programming, the clear image of a blueprint should come to mind. This is due to the fact that classes serve as the blueprints for objects. In class-based languages like Java, C++, and Python, classes define things like the data attributes and the methods/functions that'll be used to manipulate said data.

Let's take the example of a Person class. Check out the image to see how it would be declared in Java.

Typically the first thing defined and/or instantiated within the class itself are its instance data, or attributes. For example, a Dog class might have attributes for their name, behavior, A good practice is to attach the "private" access modifier keyword to the attributes so that they can only be accessed/modified via the built-in methods; meaning they can't be changed or accessed directly. On top of this, the named for the instance data should follow camelCase or snake_case notation.

Our Person class can have attributes for many things, but let's just go with a name and weight. The weight is in pounds and should be some sort of floating-point number, while the name should be a String.

The constructor(s) are an essential part of classes as they define how they should be instantiated. This goes hand-in-hand with the concept of instantiation

OOP Principles

Indeed Article
The fact that we can group related properties and the methods that perform operations on them into one unit. A class is "well-encapsulated" if its properties aren't directly accessible and can only be modified using its methods. Encapsulation ensures that the class/object code doesn't leak into the rest of the program, so it takes you one step further from spaghetti code.
When some of the properties or methods of a class are purposely hidden from the outside (usually declared as private) in order to achieve a simpler interface for both the programmers and the end-user, and less of an impact when it comes to changes involving the aforementioned hidden data. It's easier to use an object with fewer properties and methods; even if using the methods is comparable to throwing information into a magic box and getting the desired result out.
