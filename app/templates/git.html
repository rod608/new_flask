Git & GitHub

GitHub Docs: Repositories

Time to learn about Git, GitHub, & Repositories. Git is a version control software, GitHub is the company that developers make use of cloud-based Git repositories, and Repositories are the storage locations for your previous software archives in your version control process (where your projects are stored)!

Alright, so what is version control? Just as it sounds, it's a safe way to save changes in your code over periods of time without having to overwrite previous versions; in case you have to go back to it later on for some reason. Git counts as distributed version control, meaning each individual developer working with a repository has a copy of it; this is better as centralized as that one entails the "main" project being somewhere specific. Overall, you can expect to initialize or clone a repository, add your changes, commit a snapshot of your code, and push (if you're working alone on a small project, that is)!

Commits

GitHub Docs: Commits

A commit is can be compared to a snapshot of your repository at a specific time.

Commits are made after implemented major changes, or even minor ones that hold some sort of value. The point of implementing version control at all is being able to go back to a previous state if things were to go wrong down the line. Of course, you have to add (git add) the files you've updated to the stage before attempting to commit. You can commit with or without a message, but it's good practice to include one.

Branches

GitHub Docs: Branches

Instead of pushing straight to main, create a branch. Branches are contained areas of your repository that allow for safely modifying code before without making changes to the "main" or "master" branch.

We all know what "pushing" is in GitHub, although there's a good chance that you've only experienced pushing to "main" or "master", which are branches themselves. In reality, this is an awful habit. Think of it this way: When working with a team of developers, everyone will be delegated their own "feature" to implement. Let's say a team member completed their feature, but they unknowingly pushed to the "main" or "master" branch. In that scenario, the possibility of a mass code breakage arises. The predicament could have been easily avoided if the team member had created a new branch and tested his work from there, before eventually merging to the main branch. Let's get into merging:

Merging

GitHub Docs: Merging

After testing your code in the sub-branch, you're ready to merge. Merging entails piecing together the sub-branches back into the main one, which unifies the code once again.

Ok, so merging isn't as clear as I made it to be above. You may not be ready to merge after confirming your code is safe as it's best to fetch remote commits, and possibly implement a fast-forward merge instead of a regular one.

Tags

GitHub Docs: Tags

Tags in GitHub are similar to comments made in code; they're useful for annotation purposes.

As tags are associated with commits, you can think of them as a different form of "message" attached to the commit. If you use one, it should be useful information. For Example, you may have a tag to signify a new version of your software; whether it be major or minor. It's just like those Windows updates you see often; they have specific "version" names if you look carefully in settings.
